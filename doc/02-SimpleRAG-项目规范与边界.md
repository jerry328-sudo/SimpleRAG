# SimpleRAG 项目规范与边界

## 1. 文档目的

本文档用于定义 `SimpleRAG` 项目的硬约束、实现边界、演进边界和评审标准。

这不是产品设计文档，也不是实现计划文档。  
它的作用是回答以下问题：

- 哪些需求属于项目范围，哪些不属于。
- 哪些实现方式是允许的，哪些实现方式必须避免。
- 后续新增功能时，什么叫“沿着既定方向扩展”，什么叫“偏离项目目标”。
- 代码评审、需求评审和发布前检查时，应该按什么标准判断是否合格。

如果后续实现与本文件冲突，优先做法是：

1. 先确认是否真的需要变更边界。
2. 如果需要，先更新本文件和主设计文档，再修改代码。
3. 不允许“先改实现，后补边界说明”。

## 3. 项目一句话定义

`SimpleRAG` 是一个 **本地持久化索引 + 远端模型 API + Obsidian 常驻侧边栏搜索/聊天** 的插件。

这个定义包含三层约束：

1. 索引是本地的，不是云端托管知识库。
2. 模型能力是远端 API 驱动，不以内置本地大模型为第一版目标。
3. 交互中心是 Obsidian 内部的侧边栏搜索与基于证据的聊天，不是独立聊天应用。

## 4. 产品边界

### 4.1 当前明确支持的对象

- Markdown 笔记
- 被 Markdown 笔记引用的图片

### 4.2 当前明确不支持的对象

- PDF
- Word、PPT、Excel
- Canvas
- 音频、视频
- 未被 Markdown 笔记引用的图片
- 任意外部网页内容

### 4.3 第一版检索能力边界

必须支持：

- 文搜文
- 文搜图
- 图搜图
- 图搜文

前提：

- 必须处于多模态模式
- 当前 embedding 模型确实支持文本与图片共空间或可对齐空间检索

如果所选模型不满足该前提，则可以降级功能，但必须显式提示，不能静默给出误导性结果。

### 4.4 第一版聊天能力边界

聊天能力不是“对整个 vault 任意聊天”，而是：

- 基于当前搜索结果继续追问
- 基于检索证据回答
- 必须带引用

以下行为不应出现：

- 不检索直接聊天
- 忽略当前搜索结果直接自由发挥
- 给出无来源回答
- 用模型补全用户 vault 中不存在的“猜测性事实”

## 5. 数据边界

### 5.1 路径边界

项目内所有运行时索引记录必须使用 vault 相对路径。

硬性要求：

- 不保存绝对路径
- 不保存盘符
- 不保存用户系统用户名
- 路径统一使用 `/`

不允许为了省事，在数据库、缓存或日志里混入绝对路径。

### 5.2 数据存储边界

运行时数据必须放在 Obsidian 插件私有目录中。

允许的目录：

```text
.obsidian/plugins/<plugin-id>/
  data.json
  storage/
  cache/
```

不允许的做法：

- 把数据库写到 vault 普通笔记目录
- 在仓库根目录生成运行时数据库
- 把缓存写到用户笔记树中

### 5.3 本地与远端的边界

本地保存：

- 配置
- 索引元数据
- chunk
- 图片引用关系
- embedding
- 索引状态

远端发送：

- 文本 chunk
- 图片内容
- 搜索 query
- rerank 候选
- 聊天上下文

要求：

- 必须在设置页或文档中清楚披露发送内容
- 不允许隐藏式遥测
- 不允许发送不必要的 vault 内容

---

## 6. 索引边界

### 6.1 索引范围边界

文本索引范围：

- 只索引 `.md`

图片索引范围：

- 仅限被 Markdown 笔记引用的图片

明确禁止：

- 索引未被引用图片
- 偷偷把 PDF、Canvas、附件文本解析加进第一版索引
- 在没有文档说明和设置披露的前提下扩大索引范围

### 6.2 文本切块边界

固定采用：

- 标题层级 + 段落块

第一版明确不做：

- 固定字数生硬切块作为主策略
- 代码块 embedding
- 为了追求召回率而把整篇笔记都当成单一 chunk

### 6.3 图片索引边界

图片不是独立知识源，而是“被笔记上下文引用的资源”。

因此必须保留：

- 图片资源记录
- 图片被哪些笔记引用
- 图片在什么标题路径下被引用

禁止的简化方案：

- 只存图片向量，不存引用关系
- 把图片硬绑定到单一笔记
- 命中图片时无法反查其来源笔记

### 6.4 变更跟踪边界

变更跟踪主机制必须是：

- Obsidian vault 事件
- 启动扫描 / 手动扫描

不允许把 Git 作为主变更跟踪机制。

原因：

- 并非所有 vault 都用 Git
- Git 不表示运行时真实内容状态
- Git 不适合索引 dirty 判断

### 6.5 索引更新边界

第一版日常更新模式是：

- 手动扫描
- 手动更新索引
- 手动全量重建

明确不做：

- 每次修改文件都自动重建 embedding
- 用户无感知地持续后台全量同步

事件监听的职责是：

- 标记 dirty

不是：

- 立即发起 embedding 请求

### 6.6 索引一致性边界

以下一致性要求必须长期成立：

1. 笔记重建索引时，旧 chunk embedding 必须清理。
2. 图片修改后，其旧 embedding 必须失效并重建。
3. 图片失去全部引用后，必须从索引中移除。
4. 搜索只能使用当前 provider/model/index mode 下有效的 embedding。

任何破坏上述四点的实现都应视为严重缺陷。

---

## 7. 搜索边界

### 7.1 搜索流程边界

搜索链路固定为：

1. query embedding
2. 向量召回
3. rerank
4. 结果展示

其中 rerank 可以关闭，但不能把“聊天模型直接决定搜索结果”当成主搜索实现。

### 7.2 结果呈现边界

默认结果视图为：

- `All`
- 并提供 `Notes` / `Images` 筛选

必须展示的核心信息：

- 类型
- 路径
- 标题路径
- 摘要

图片结果还必须能说明：

- 它被哪些笔记引用

### 7.3 搜索质量边界

以下做法必须避免：

- 混用不同 embedding 模型的向量进行同一轮召回
- 在 `text-only` 模式下召回图片向量
- 明知索引配置已失效仍继续给结果而不提示重建

---

## 8. 聊天边界

### 8.1 聊天上下文边界

聊天上下文来源必须是：

- 当前搜索结果派生出的 document packets

不允许：

- 直接把全库所有内容塞给模型
- 绕过搜索直接全局问答

### 8.2 文档级上下文边界

检索单元是 chunk，聊天上下文单元是 document packet。

规则：

- 短笔记可直接用全文
- 长笔记只用命中区域扩展窗口

这里的原则是：

- 聊天不只看单个 chunk
- 但也不默认总是整篇全文

### 8.3 引用边界

聊天输出必须带引用，而且引用必须可追溯到原始证据。

允许：

- 行内引用编号
- 底部来源卡片

不允许：

- 只说“参考了若干文档”但不给具体来源
- 用扩展上下文替代原始证据定位

文本引用应回到：

- chunk 或其标题路径位置

图片引用应回到：

- 图片资源
- 以及其引用位置

### 8.4 图片证据边界

如果搜索结果中存在高相关图片，聊天链路必须能够：

- 将图片转成可供模型理解的证据表示
- 保留图片与笔记引用关系
- 在最终引用中标记为图片证据

不允许在 UI 上展示图片检索能力，却在聊天阶段把图片证据完全丢弃。

---

## 9. Provider 边界

### 9.1 模块化边界

以下能力必须通过 provider 抽象接入：

- embedding
- rerank
- chat

不允许：

- 在业务代码中直接写死某家厂商 HTTP 协议细节
- 在搜索、索引、聊天逻辑中直接分支判断某个具体平台品牌

### 9.2 配置变更边界

以下变更必须导致索引失效并要求重建：

- embedding provider
- embedding model
- image embedding 开关

以下变更不要求重建：

- rerank provider/model
- chat provider/model
- rerank/chat 的 base URL、token

### 9.3 能力声明边界

provider 必须声明其能力，而不是让调用方靠猜。

至少要能判断：

- 是否支持文本 embedding
- 是否支持图片 embedding
- 是否支持跨模态
- 是否支持 rerank
- 是否支持聊天

如果某项能力不支持，必须在设置、索引或搜索阶段显式报错或禁用，而不是静默失败。

---

## 10. UI 边界

### 10.1 主入口边界

第一版主入口是常驻侧边栏 View。

允许：

- 通过命令面板或 Ribbon 打开侧边栏

不建议：

- 把搜索主体验做成一次性 Modal
- 新增多个互相重叠的入口

### 10.2 设置页边界

设置页默认暴露应尽量克制。

默认区只放：

- Embedding provider
- Embedding model
- Base URL
- API token
- Enable image embedding
- Enable AI chat
- Index status
- 常用索引操作

其他大多数参数放进 `Advanced`。

禁止：

- 为了“可配置”把大量内部参数默认暴露给普通用户
- 把 chunking、索引一致性、引用关系这类核心实现细节做成随意可改的设置

### 10.3 状态反馈边界

用户必须明确知道：

- 当前是 `text-only` 还是 `multimodal`
- 当前索引是否最新
- 是否有 dirty 文件
- 是否需要全量重建

不允许让索引失效状态处于不可见状态。

---

## 11. 存储实现边界

### 11.1 当前允许的过渡实现

如果当前阶段为了快速迭代，存储层暂时使用：

- JSON 持久化
- 或其他轻量方案

可以接受，但必须满足两个条件：

1. 对外抽象保持数据库语义一致。
2. 不让上层业务依赖某种临时存储细节。

### 11.2 最终演进方向

目标仍然应是：

- 真正的本地 SQLite 或等价持久化数据库能力

原因：

- 数据关系复杂
- 状态迁移需求明确
- 后续查询、清理、迁移更稳定

### 11.3 禁止的存储做法

- 让 UI 直接操作原始存储文件
- 让上层逻辑绕过存储抽象自行读写索引结构
- 多处各自维护一套索引真相

索引真相必须单一。

---

## 12. 测试与验证边界

以下类型的行为必须有自动化测试覆盖：

1. Markdown 解析与 chunking
2. 图片引用解析
3. 向量相似度与召回过滤
4. 索引重建时旧 embedding 清理
5. 图片修改 / 删除 / 失去引用后的索引一致性
6. 聊天上下文构造，尤其是图片证据路径

仅靠以下手段不算充分验证：

- `npm run build`
- 手工点几次 UI
- 看控制台没报错

### 12.1 每次提交前最低验证标准

至少应满足：

- 能构建
- 相关单元测试通过
- 改动涉及索引、搜索或聊天时，新增或更新回归测试

---

## 13. 代码评审边界

代码评审时，优先检查以下问题，而不是先看风格：

1. 是否破坏索引一致性
2. 是否扩大了产品边界却没有更新文档
3. 是否把不该发送的数据发到远端 API
4. 是否混用了不同 provider/model 的 embedding
5. 是否让图片引用关系丢失
6. 是否让聊天回答失去引用能力

以下情况应视为高优先级问题：

- 搜索结果可命中过期数据
- 切模型后仍使用旧向量
- 图片删除后仍可搜索到
- 聊天结果无法追溯来源
- 代码把绝对路径写入持久化层

---

## 14. 需求变更边界

以下类型的需求变更可以视为“边界内扩展”：

- 增加更多 provider
- 增加更好的 rerank
- 增加更稳的引用定位
- 增加更好的图片上下文表达
- 将 JSON 持久化迁移为 SQLite/WASM

以下类型的需求变更视为“边界外扩展”，需要重新评审并更新主设计文档：

- 支持 PDF/Office 文档
- 支持 OCR
- 支持全 vault 自动后台实时 embedding
- 支持云端托管索引
- 支持无搜索前置的全局聊天
- 支持跨 vault 联合索引

---

## 15. 发布边界

在准备对外发布或给他人试用前，至少要确认：

- 不会把绝对路径持久化
- 不会静默把超出预期的内容发送到远端
- 设置页能明确看出启用了哪些远端能力
- 索引失效时会提示而不是默默继续使用旧索引
- 删除或重建索引不会留下脏结果

---

## 16. 开发执行清单

后续实现任何功能前，都应先过一遍这份清单：

### 16.1 产品层

- 这个功能是否仍在当前边界内？
- 是否扩大了索引对象范围？
- 是否改变了聊天上下文边界？

### 16.2 数据层

- 是否引入了绝对路径？
- 是否破坏了索引单一真相？
- 是否会留下过期 embedding？

### 16.3 API 层

- 是否通过 provider 接入？
- 是否有能力声明？
- 是否会额外发送未披露数据？

### 16.4 UI 层

- 是否会增加普通用户默认暴露的复杂度？
- 是否会让索引状态变得更不透明？

### 16.5 测试层

- 是否新增了回归测试？
- 是否覆盖了失败路径？

---

## 17. 当前阶段的最终约束总结

可以把本项目当前边界压缩成以下 12 条硬规则：

1. 只索引 Markdown。
2. 图片只索引被 Markdown 引用的资源。
3. 不保存绝对路径。
4. 运行时数据只放插件私有目录。
5. 变更跟踪靠 Obsidian 事件与扫描，不靠 Git。
6. 日常更新是手动增量更新，不做无提示自动重建。
7. 搜索只能使用当前模型的有效 embedding。
8. 切 embedding 模型或模式必须要求重建索引。
9. 聊天必须基于当前搜索结果。
10. 聊天必须带引用。
11. 图片证据不能在聊天链路里丢失。
12. provider 必须模块化，业务逻辑不得写死厂商。

---

## 18. 结论

本文件的核心目标不是限制开发速度，而是防止项目在实现过程中逐步滑向另一个产品。

`SimpleRAG` 的第一版不是万能知识助手，不是全库自由对话系统，也不是通用文档解析平台。  
它是一个 **围绕 Obsidian Markdown 笔记构建本地 RAG 索引，并通过远端 API 提供可追溯搜索与聊天能力** 的插件。

只要后续实现持续满足这个定义，就仍然在项目边界内。
