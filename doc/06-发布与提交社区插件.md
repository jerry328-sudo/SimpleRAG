# 发布与提交社区插件

发布分为两步：先做 GitHub 发布，再可选提交官方社区插件列表。

## 1. 本地发布准备

每次发布前确认：

- `package.json` 版本
- `manifest.json` 版本
- `manifest.json` 的 `minAppVersion`
- `versions.json` 是否同步
- README / LICENSE / `manifest.json` 是否都在仓库根目录

版本号遵循 `x.y.z`，通常不带 `v` 前缀。

## 2. 生成发布文件

官方流程会上传到 GitHub Release 的附件通常有：

- `main.js`
- `manifest.json`
- `styles.css`（存在则上传）

`main.js` 来自构建产物，不是源码文件。

注意：

- `manifest.json` 既要存在于仓库根目录，也要作为 Release 附件上传
- `main.js` 不建议提交到仓库源码分支，但必须出现在 Release 里

## 3. 标签和版本一致性

发布标签必须与 `manifest.json.version` 对齐，例如：

- `manifest.json` = `1.2.0`
- release tag = `1.2.0`

## 4. 为什么 `versions.json` 不总改

当你只改代码、未改最低兼容版本时，不需要更新映射。

当你把 `minAppVersion` 提高时，才更新 `versions.json` 以保证旧版用户仍能下载安装合适版本。

## 5. 自动化发布（可选但推荐）

官方有一篇单独的 GitHub Actions 指南，核心思路是：

1. 在 `.github/workflows/release.yml` 中监听 tag push
2. 用 Node 18 构建插件
3. 自动创建 draft GitHub Release
4. 自动上传 `main.js`、`manifest.json`、`styles.css`

这样做的好处是：

- 避免手工漏传附件
- 避免 tag 与 release 内容不一致
- 提高重复发版的稳定性

## 6. 提交社区插件目录

官方说明里明确写了：

- 只需要提交第一次发布的版本
- 后续插件更新由 GitHub Release 直接分发给用户

提交前，仓库根目录至少有：

- `README.md`
- `LICENSE`
- `manifest.json`

官方目录提交流程：

1. 在 `community-plugins.json` 中加入条目
2. 提交 PR
3. 等待验证机器人与人工审核

条目通常包括：

- `id`
- `name`
- `author`
- `description`
- `repo`

额外约束：

- `id` 不能包含 `obsidian`
- `repo` 要写成 `owner/repo`
- `id`、`name`、`author`、`description` 应与 `manifest.json` 对应字段保持一致

## 7. 提交中常见坑

- `description` 规则不一致（长度、标点）
- `repo` 名称格式不对
- `manifest.json` 与 release 附件不一致
- README / LICENSE 不完整
- 未披露联网、付费、数据行为
- 仍然保留了 sample plugin 的类名、命令名、示例代码
- `isDesktopOnly` 设置错误
- 命令 ID 手动带了插件 ID

## 8. `description` 的官方提交要求

官方 `Submission requirements for plugins` 对描述的要求比很多人想象中严格：

- 最长 250 个字符
- 语义清楚，最好用动作开头
- 末尾用句号 `.`
- 不要写 “This is a plugin...”
- 不要用 emoji 或花哨符号

更好的写法像这样：

- `Translate selected text into English.`
- `Generate notes automatically from web articles.`
- `Sync highlights and annotations from Kindle.`

## 9. 审核反馈处理

官方常见反馈会直接在 PR 下给。建议：

- 按评论逐条修复
- 推送版本更新
- 在 PR 中回复处理完成

官方还特别提醒：

- 不要新开一个 PR
- 即使 GitHub 提示该 PR 有 merge conflicts，也先不要 rebase 或自行处理，等官方在合并时解决

## 10. 合规与安全常识

官方政策要求内容透明：

- 不允许有隐蔽行为或捆绑自更新
- 不做客户端遥测
- 代码与依赖许可符合要求
- 不得混淆导致行为不可理解

## 11. Node / Electron 与桌面专用声明

官方提交要求再次强调：

- 如果用到 Node.js 或 Electron API，`manifest.json.isDesktopOnly` 必须为 `true`

例如这些通常都意味着桌面专用：

- `fs`
- `path`
- `crypto`（若使用 Node 版本）
- `electron`

## 12. 推荐上架前最后核对

- `manifest.json` 字段正确
- README 能说明权限和边界
- 命令/设置/功能都能在最小场景运行
- 发布附件齐全
- release notes 写清更新内容
- `description` 符合 250 字内、句号结尾、无 emoji
- 若使用桌面 API，`isDesktopOnly` 已正确设置
