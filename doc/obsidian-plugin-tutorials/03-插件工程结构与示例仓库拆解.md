# 插件工程结构与示例仓库拆解

官方样例仓库是 `obsidian-sample-plugin`，文件名和职责可以直接作为你新项目的最初结构基线。

仓库地址： https://github.com/obsidianmd/obsidian-sample-plugin

## 1. `manifest.json`

插件清单，决定安装识别和元信息。常见字段：

- `id`
- `name`
- `version`
- `minAppVersion`
- `description`
- `author`
- `authorUrl`
- `fundingUrl`
- `isDesktopOnly`

建议：

- `id` 与文件夹名一致
- `id` 不要包含 `obsidian`
- 有桌面专属能力时 `isDesktopOnly: true`

## 2. `src/main.ts`

插件主类通常继承 `Plugin`，主要做 `onload` 注册和 `onunload` 清理。

常见注册入口：

- `addCommand`
- `addRibbonIcon`
- `addStatusBarItem`
- `addSettingTab`
- `registerEvent`
- `registerDomEvent`
- `registerInterval`
- `registerView`

## 3. `src/settings.ts`

建议把设置拆成独立文件，定义：

- 设置接口（类型）
- 默认值对象
- 设置页类（`PluginSettingTab`）
- UI 变化后的 `saveData`

这样主入口不会变成“大泥球”。

## 4. `styles.css`

定义插件自有样式。官方示例把 UI 样式和交互分离放在这里，不建议在 JS 中直接拼接样式字符串。

## 5. `package.json`

定义构建脚本、依赖和模块版本。典型脚本是：

- `dev`：监听编译
- `build`：生产构建
- `version`：版本同步
- `lint`：静态检查

`obsidian` 依赖通常放在 `dependencies` 用于类型和构建时引用。

## 6. `tsconfig.json`

TypeScript 严格参数配置，减少运行时 bug：

- 严格模式相关选项
- `moduleResolution`
- `target` / `lib` 配置

## 7. `esbuild.config.mjs`

核心打包器配置，决定 `main.ts` 怎么输出成 `main.js`。关键点：

- `src/main.ts` 作为入口
- 输出为 Obsidian 可加载文件
- 常见 external 规则避免重复打包宿主模块

## 8. `versions.json`

在提高 `minAppVersion` 时，维护该映射可以兼容旧版用户。只在必要时更新。

示例：

```json
{
  "1.0.0": "1.5.0",
  "1.1.0": "1.6.0"
}
```

## 9. 其他脚本文件

示例仓库常见有版本辅助脚本：

- `version-bump.mjs`

用来同步 `package.json` 与 `manifest.json` 的版本号，并按需求同步 `versions.json`。

## 10. 你可直接复用的目录演进方式

小插件：

```text
src/
  main.ts
  settings.ts
manifest.json
styles.css
```

中大型插件：

```text
src/
  main.ts
  settings.ts
  commands/
  views/
  modals/
  services/
  utils/
manifest.json
styles.css
versions.json
```

### 设计原则

- 单一职责，主入口只做组织装配
- 功能模块分文件，便于维护与测试
- 样式与逻辑分离

## 11. 推荐与官方一致的本地命名策略

目录名与清单 id 保持一致。例如：

```text
obsidian-vault/
└─ .obsidian/plugins/obsidian-plugin-starter/
   ├─ main.js
   ├─ manifest.json
   ├─ styles.css
   ├─ versions.json
   └─ src/
```

这样能减少 Obsidian 识别错误和提交审核差错。
