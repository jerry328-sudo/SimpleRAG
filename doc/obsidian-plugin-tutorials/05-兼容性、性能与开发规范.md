# 兼容性、性能与开发规范

插件在本地能跑，并不代表它：

- 在移动端能跑
- 在大型 vault 中性能稳定
- 在新版 Obsidian 中不出兼容问题
- 能顺利通过社区目录审核

下面这篇把官方 `Mobile development`、`Optimize plugin load time`、`Defer views`、`Plugin guidelines` 的高频规则合并成一张开发检查表。

## 1. 移动端开发基础

官方移动端文档核心结论：

- 不要默认假设桌面能力在移动可用。
- 判断平台后再分支。
- 某些 Electron/Node API 不应在移动端触发。
- 自定义状态栏项在移动端不受支持。

### 平台判断

```ts
import { Platform } from 'obsidian';

if (Platform.isMobile) {
  // 移动端逻辑
}

if (Platform.isIosApp) {
  // iOS 逻辑
}

if (Platform.isAndroidApp) {
  // Android 逻辑
}
```

### 模拟移动端

可用 `this.app.emulateMobile(true)` 做快速验证，但真机验证更可靠。

## 2. `isDesktopOnly`

如果插件依赖桌面专属能力（包括 Electron、Node API），`manifest.json` 要设置：

```json
{
  "isDesktopOnly": true
}
```

否则可能在移动端产生异常行为。

## 3. 移动端额外避坑

官方明确强调这些点：

- 不要在顶层直接 `import fs`、`import path`、`import electron`
- 需要桌面专属能力时，用 `Platform.isDesktopApp` 分支后再动态 `require()`
- 不要用 `process.platform` 判断平台，改用 Obsidian 的 `Platform`
- 需要网络请求时优先用 `requestUrl`，不要默认用 `fetch` 或 `axios.get`
- 不要把 `Vault.adapter` 直接断言成 `FileSystemAdapter`，先做 `instanceof` 检查

### 正则 lookbehind

官方还特别提醒：

- 如果你要兼容较老的 iOS，避免依赖正则 lookbehind
- iOS 16.4 之前对它支持不完整

## 4. 性能：启动不应变慢

官方推荐 `onload` 保持轻量。建议只做注册类操作。

- 不要在 `onload` 做大量文件遍历
- 不要在 `onload` 发起重网络请求
- 不要做重计算和长轮询初始化
- 发布版应使用生产构建与压缩后的 `main.js`

建议把非关键初始化放在 `onLayoutReady`。

## 5. Vault 事件与启动时机

`vault` 的 create/modify 事件在启动阶段可能出现“系统性”触发，若你在启动早期注册监听，可能误判行为。建议：

- 在布局就绪后再注册相关监听
- 或在回调中检查上下文再执行

示例方向：

```ts
this.app.workspace.onLayoutReady(() => {
  this.registerEvent(this.app.vault.on('create', this.onFileCreate));
});
```

## 6. 自定义视图与延迟加载

从 Obsidian `1.7.2` 起，工作区里的 view 可能先以 `DeferredView` 的形式存在。官方建议：

- 访问 `leaf.view` 时先做 `instanceof`
- 不要假设 `getLeavesOfType()` 拿到的 `leaf.view` 已经是你自定义的真实视图
- 如果你真的要操作某个视图实例，先 `await workspace.revealLeaf(leaf)`
- 极少数场景才使用 `leaf.loadIfDeferred()`，因为它会取消掉这项性能优化

示例：

```ts
const leaf = this.app.workspace.getLeavesOfType(MY_VIEW_TYPE).first();

if (leaf) {
  await this.app.workspace.revealLeaf(leaf);

  if (leaf.view instanceof MyCustomView) {
    // 安全访问
  }
}
```

## 7. 安全与实现风格

官方插件规范偏好“声明式与安全”：

- 不要用 `innerHTML`、`outerHTML`、`insertAdjacentHTML` 直接拼接用户输入
- 避免直接操作未净化的外部输入
- 尽量用 `createEl`、`createSpan`、`createDiv` 等 API 组织 DOM
- 事件、定时器都使用 `register*` 机制
- 不要把样式写死在 JS 里，优先使用 CSS 类与 Obsidian CSS 变量

## 8. UI 与文案规范

官方指南强调可读性与体验：

- 命令名简洁、清晰，避免重复插件名
- 命令 ID 不要重复包含插件 ID，Obsidian 会自动前缀
- 不默认设置快捷键
- 设置页不建议过度模板化标题
- 设置文案用自然语言，避免全部大写或“炫技”命名
- UI 文案使用 sentence case
- 设置页标题优先用 `setHeading()`，不要手写 `<h1>` / `<h2>`

## 9. 网络、账户、权限披露

官方提交要求中要求透明披露：

- 是否联网
- 使用第三方服务
- 是否需要账号
- 是否收费
- 是否访问外部文件
- 是否有遥测

建议都写在 README 显著位置。

## 10. API 使用风格的官方偏好

这些是审核里非常高频的意见：

- 用 `this.app`，不要用全局 `app`
- 编辑当前活动文件时优先用 `Editor`，不要直接 `Vault.modify`
- 后台改文件优先用 `Vault.process`
- 修改 frontmatter 优先用 `FileManager.processFrontMatter`
- 删文件优先走回收站逻辑，不要直接 `vault.delete`
- 优先使用 Vault API，不要默认下沉到 Adapter API
- 根据路径找文件时，使用 `getFileByPath` / `getFolderByPath` / `getAbstractFileByPath`
- 接收用户输入路径时使用 `normalizePath()`

## 11. 审核前检查清单（摘要）

- manifest 信息是否准确
- `description` 长度、语法是否规范
- `isDesktopOnly` 是否正确
- 默认快捷键是否为空
- 无明显安全风险 DOM 操作
- 所有 `onload` 注册都可回收
- 文档披露完整
- 无 sample plugin 残留命名与演示代码
- 没有不必要的 `console.log`

## 12. 性能与兼容建议汇总

1. 把重任务延后到 `onLayoutReady`
2. 用 `Vault.process` 做批量安全改写
3. 预期移动端行为时加条件分支
4. 使用 `requireApiVersion` 做版本门控
5. 避免在启动阶段做高耗时扫描
6. 访问 View 实例时做 `instanceof` 检查
7. 用 CSS 类与变量，不在 JS 里硬编码样式
